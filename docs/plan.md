> Phased build plan. **Check the "Current phase" marker before adding functionality.** If a thing belongs to a later phase, don't build it yet — flag it instead. Scope creep is this project's primary risk. Refine phases as you learn; keep this file the source of truth for sequencing. Field-test reviews in `docs/snapshots/` feed the **Discovered / unscheduled** backlog near the bottom — check there for insights not yet folded into a phase.

> **What "good" means:** `docs/product-requirements.md` is the fidelity bar (tiered requirements R1.1–R6.4 + the five load-bearing tensions); `docs/acceptance-testing/fidelity-criteria.md` turns it into pass/fail gates. The phases below sequence the work; those two files define when it's faithful.

Phases 1–7 are shipped/closed. The plan was re-cut on 2026-07-10 with the owner: the old catch-all "Phase 7 — Later/optional" was split into **Phase 7 — post-launch polish** (finish what a public visitor/reader now touches — now done), **Phase 8 — validation & signal quality** (the evidence push, V1 first — the current focus), and **Phase 9 — later/post-traction** (parked until the drafting habit demonstrably takes hold). Earlier re-cut history: on 2026-06-17 the experience/signal-quality/validation work that had accreted in Phase 5 became Phase 6, and the old "Phase 6 — Later/optional" became Phase 7.)

> **Routing legend.** Open milestones below carry an annotation `— <readiness> <complexity> · <agent>` so it's clear what's ready, how hard, and who should build it. Completed (`[x]`) lines are left unannotated.
>
> - **Readiness:** 🟢 fully defined, ready to build · 🟡 mostly defined, decisions along the way · 🟠 not defined, needs pre-work/planning · 🔴 concept only, no design.
> - **Complexity:** Low · Med · High.
> - **Agent:** 🧠 capable/expensive (judgment, design, prompt-quality, architecture) · ⚙️ mid · 🔧 simple/mechanical (well-specified, low-decision).
>
> **This is required metadata, not decoration.** Every new open milestone (here or in any phase section) gets the annotation **on creation** — there is no unannotated open item. And it's **living:** whenever work touches an item, re-assess and update its annotation in the same change. **Readiness** moves the most (e.g. a design spec lands → 🟠→🟡, or a milestone is fully specced → 🟡→🟢); **complexity** and **agent** shift less often but get re-rated when new information changes the picture (e.g. a "simple" item turns out to need architectural judgment → 🔧→🧠). When an item is completed, drop the annotation as the `[x]` is added.

> **Milestone anatomy — the default entry style.** The **Phase 6** entries (now in [`plan-archive.md`](plan-archive.md)) are the template: a plan item is a **living record of a decision**, not a one-line to-do. Write new milestones — and grow existing ones as work lands — from as many of these parts as apply, in this order:
>
> 1. **Bold title**, carrying any tracking id(s): `**Widen the section check to same-section pairs (OBS-033/UX-018)**`.
> 2. **Provenance** — how and when it surfaced, as an **absolute date**: `field-discovered 2026-07-08`, `decided 2026-06-18 (with the user)`, `field-observed (owner)`.
> 3. **The problem** — what's wrong and why it matters, concretely: name the real session/example that exposed it and the `file.ts:line` if known.
> 4. **The decision** — a **bold state marker + date** naming the chosen mechanism (and the instructive rejected alternative): `**Decided 2026-07-09 (mechanism A):** …`. The markers trace an arc a live entry walks: **Field-discovered → Decided / Design settled → Shipped**.
> 5. **Implementation specifics** — the exact files, functions, and tokens in `code`, so the entry is self-locating without opening the PR.
> 6. **Verification** — the guard that keeps it fixed: the ratchet fixture, unit test, or live-verify.
> 7. **Cross-references** — trailing `→ see docs/projects/<file>.md` (the design home) and `· resolves OBS-NNN / UX-NNN`.
> 8. **Routing annotation** (open items only) — the four routing axes defined by the legend above, appended as `— <readiness> <complexity> · <agent>` plus, in a parallel-lane phase, `· Lane: <name>`. Spell out all four:
>    - **Readiness** — how defined / ready-to-build the work is: 🟢 ready · 🟡 mostly, decisions along the way · 🟠 needs pre-work/planning · 🔴 concept only.
>    - **Estimated size** (complexity) — Low · Med · High (compound forms like `Med–High` or `High-decision/Low-build` are fine).
>    - **LLM capacity** (agent) — who should build it: 🧠 capable/expensive (judgment, design, prompt-quality, architecture) · ⚙️ mid · 🔧 simple/mechanical, low-decision.
>    - **Lane** — `· Lane: <name>`; it names the code territory the item lives in (see the _Parallel work lanes (live)_ legend below; the original Phase-6 lane table + rules of the road are in [`plan-archive.md`](plan-archive.md)).
>
>    The annotation is **required on every open item and living** — re-rate it whenever work touches the item — and **dropped** when the item is checked `[x]`.
>
> Not every item needs all eight — a trivial mechanical fix can stay terse — but the **default** is the fuller Phase-6 form, and a completed item **keeps** its record inline (see the archived Phase 6) rather than being deleted. Keep any multi-line notes or example lines inside a blockquote (like this one) so the routing-annotation contract doesn't read them as milestones.

> **Parallel work lanes (live).** The lane axis names the **code territory** an item lives in, so concurrent sessions don't collide on a hub file. The six Phase-6 lanes still apply and keep their definitions in [`plan-archive.md`](plan-archive.md) → _Parallel work lanes_ (**Editor** · **Feed UI** · **Prompt/signal** · **Validation** · **Lifecycle** · **Visual**, with the hub-file table and rules of the road). Post-launch work added four territories (2026-07-10):
>
> - **Model/router** — `src/model/*`: provider adapters, pools/cool-down, capability descriptor, debug log. Shares little with the eval lanes; parallel-safe with most.
> - **Site/docs** — README, `docs/`, site legal pages, spec-writing. No `src/`; parallel-safe with everything.
> - **Ops** — CI workflows, `scripts/`, GitHub settings, the release process. No `src/`; parallel-safe with everything (but only one Ops change to the release path at a time).
> - **Platform** — app-shell/packaging territory: Tauri wrapper, serverless proxy, external integrations, monetization plumbing. Nothing here is active; re-rate when an item is scheduled.
>
> Same rules as Phase 6: two items are parallel-safe iff they're in different lanes and neither is Visual; items sharing a lane's hub file serialize within the lane.
>
> **Enforced:** `docs/plan.annotations.test.ts` requires a `· Lane: <name>` (from this list) on every open milestone inside a "Phase N" section, alongside the readiness/agent markers — added 2026-07-10 after the plan re-cut dropped the lane axis unnoticed (the axis existed only as prose, so nothing failed). Discovered/unscheduled ideas stay un-laned by convention. If you add a lane name here, add it to the test's `LANE_NAMES` in the same change.

This is **not** guardrail territory: no user judgement is being overridden, it is the same engine's own stale output, and under engine exclusivity "yesterday's pass left junk" becomes the common case rather than an edge one. Weigh against why the check is per-session at all — attribution and revoke both scope on `sessionId`, and a cross-session close would let a fresh pairing clear cards the user may not associate with it. _(4) Pin the hard wall with a test._ "An agent cannot close a card it never wrote" currently holds as an **emergent property** of two unrelated facts (the id-free snapshot, the one identity check) and is asserted nowhere. A reasonable future convenience — ids in the snapshot so the agent can reference cards precisely — removes it silently with a green suite. Same defensive logic as `AGENT_OBSERVATION_FIELDS`' own no-spread comment. **Explicitly rejected: letting the agent close native cards, or bulk-closing on a quality axis the user names ("close all low confidence").** Retract deliberately writes no `DismissalSuppression` — the module header states that an agent withdrawing its own card and a user rejecting a finding are different acts, and conflating them would mute observations the user never dismissed — so a delegated cull teaches the system nothing while removing the feed the user was going to judge. "Close the low-confidence ones" is also one rephrasing from "close the ones I don't want to hear", which is the exact machine-scale form of the silencing G1 (flattery-resistant dismissal) exists to prevent; the `user_lens` carve-out does not stretch to cover it, since muting a lens is retracting _your own request_ and unsolicited critique is not that. **The legitimate need behind the ask is served without any close permission:** the agent already sees every card, so "which 3 of these matter most, and why" is answerable **as advice in the agent's own session**, leaving the dismissals — and therefore the suppressions that teach — with the user. Worth one paragraph in the skill. **Verification guard:** a refusal test per out-of-reach population (native card · other session's card), an `agentSkillExamples`-style assertion that the retraction guidance survives the prompt rewrite, and a `closureLabel` case for whatever reason (2) settles on. **Sequencing: after engine exclusivity and the `user_lens` design pass**, both of which move the ground under it — exclusivity decides how much of the feed is the agent's own, and the lens pass owns "close all lens cards" as a lens-lifecycle question rather than a curation one. → see `docs/projects/agent_connected_eval.md` (§ Trust & attribution) · `docs/projects/user_directed_review.md` · `docs/projects/philosophy_guardrails.md` (G1) · `docs/mechanics/agent-bridge.md` (§ Retract and revoke) — 🟡 Low–Med · 🧠 · Lane: Platform

## Standing rules across all phases

1. **No fix-application affordances. Ever.** (The product principle — see `CLAUDE.md`.)
2. Observations stay within the **fixed, typed taxonomy**.
3. **No per-keystroke full-document scans;** cross-doc checks go through the claim ledger.
4. **Quiet while generating, opinionated while revising.**
5. **Local-first / privacy** — no required server, telemetry, or egress without a decision logged in this file.
