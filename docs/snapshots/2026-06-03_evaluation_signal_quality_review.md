# Snapshot: Evaluation Signal Quality Review (2026-06-03)

## What We Tried
We conducted a cold assessment of a debug log generated from a section-by-section paste test of a PRD (`docs/assets/phase1-test-text.md`). The goal was to verify the effectiveness of the recent signal-quality remediation work (`docs/projects/evaluation_signal_quality.md`) and to evaluate the overall usefulness, flow, and API usage of the resulting observations.

## What Were the Results
The Tier A/B/C fixes recently implemented are performing exceptionally well:
- **Heading Hallucinations Eliminated:** The transition to section-as-eval-unit was successful. Headings are no longer evaluated in isolation, completely removing the "section is empty" and "this is a PRD" hallucinations.
- **Ledger Cleanliness:** The claim ledger correctly filtered out meta-claims.
- **Unsupported Claim Carve-Out:** Success targets and metrics (e.g., "5-6 week estimate") were correctly identified as commitments/metrics without triggering false `unsupported_claim` flags.
- **Efficiency:** The doc-level dirty-check effectively prevented redundant evaluations. API calls resolved rapidly (fast checks ~1s, strong checks ~1.7s) with no 40s timeouts.
- **True Positive Caught:** The system successfully identified a planted contradiction (a Q3 start date vs. a Q2 launch target).

However, two regressions/misses were identified and fixed during this session:
1. **API Key Leakage:** The raw API key was still being logged in the debug payloads. This was patched by aliasing the key (`key=<free>` or `key=<paid>`) at the point of recording in `llmLogger.log`.
2. **Contradiction Confidence Hedging:** The un-hedged, hyper-confident contradiction prompt was still firing on the free tier. This occurred because `router.strong` was attempting to use the `FREE_STRONG_POOL` first, even when a paid key was provided. This routing logic was updated so the presence of a paid key routes directly to the `PAID_STRONG_POOL`, ensuring the prompt's confidence matches the model's capabilities.

## Observations and Ideas for the Future
Looking closely at the feedback generated by the AI, it acts like a brilliant but pedantic co-worker. It excels at catching missing edge cases (e.g., offline users) and chronological paradoxes, but risks eroding trust with noisy, low-value nitpicks.

**Ideas for future roadmap consideration:**
- **Domain-Specific Jargon Allow-lists:** The AI flagged standard tech terms ("soft launch", "rollout cohort") as undefined jargon. A mechanism for users to set a domain or provide a company dictionary is needed to prevent annoying false positives.
- **Tension vs. Contradiction:** The AI flagged a strategic tradeoff (notifying on fraud creates friction vs. trains bad behavior) as a hard logical contradiction. We should consider introducing a `strategic_tension` observation type so the AI has a bucket for philosophical conflicts that aren't strictly factual paradoxes.
- **Observation Aggregation:** The Q2/Q3 timeline paradox triggered three separate observation cards across different checks. In the future, the UI needs to aggregate cross-type flags on the same span into a single, high-impact card.
- **Confidence Badging:** High-impact checks like contradictions should carry visual hierarchy (e.g., a "High Impact" badge) to separate them from minor clarity nits in the feed.
