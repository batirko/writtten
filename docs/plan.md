> **Routing legend.** Open milestones below carry an annotation `— <readiness> <complexity> · <agent>` so it's clear what's ready, how hard, and who should build it. Completed (`[x]`) lines are left unannotated.
>
> - **Readiness:** 🟢 fully defined, ready to build · 🟡 mostly defined, decisions along the way · 🟠 not defined, needs pre-work/planning · 🔴 concept only, no design.
> - **Complexity:** Low · Med · High.
> - **Agent:** 🧠 capable/expensive (judgment, design, prompt-quality, architecture) · ⚙️ mid · 🔧 simple/mechanical (well-specified, low-decision).
>
> **This is required metadata, not decoration.** Every new open milestone (here or in any phase section) gets the annotation **on creation** — there is no unannotated open item. And it's **living:** whenever work touches an item, re-assess and update its annotation in the same change. **Readiness** moves the most (e.g. a design spec lands → 🟠→🟡, or a milestone is fully specced → 🟡→🟢); **complexity** and **agent** shift less often but get re-rated when new information changes the picture (e.g. a "simple" item turns out to need architectural judgment → 🔧→🧠). When an item is completed, drop the annotation as the `[x]` is added.

> **Milestone anatomy — the default entry style.** The **Phase 6** entries below are the template: a plan item is a **living record of a decision**, not a one-line to-do. Write new milestones — and grow existing ones as work lands — from as many of these parts as apply, in this order:
>
> 1. **Bold title**, carrying any tracking id(s): `**Widen the section check to same-section pairs (OBS-033/UX-018)**`.
> 2. **Provenance** — how and when it surfaced, as an **absolute date**: `field-discovered 2026-07-08`, `decided 2026-06-18 (with the user)`, `field-observed (owner)`.
> 3. **The problem** — what's wrong and why it matters, concretely: name the real session/example that exposed it and the `file.ts:line` if known.
> 4. **The decision** — a **bold state marker + date** naming the chosen mechanism (and the instructive rejected alternative): `**Decided 2026-07-09 (mechanism A):** …`. The markers trace an arc a live entry walks: **Field-discovered → Decided / Design settled → Shipped**.
> 5. **Implementation specifics** — the exact files, functions, and tokens in `code`, so the entry is self-locating without opening the PR.
> 6. **Verification** — the guard that keeps it fixed: the ratchet fixture, unit test, or live-verify.
> 7. **Cross-references** — trailing `→ see docs/projects/<file>.md` (the design home) and `· resolves OBS-NNN / UX-NNN`.
> 8. **Routing annotation** (open items only) — the four routing axes defined by the legend above, appended as `— <readiness> <complexity> · <agent>` plus, in a parallel-lane phase, `· Lane: <name>`. Spell out all four:
>    - **Readiness** — how defined / ready-to-build the work is: 🟢 ready · 🟡 mostly, decisions along the way · 🟠 needs pre-work/planning · 🔴 concept only.
>    - **Estimated size** (complexity) — Low · Med · High (compound forms like `Med–High` or `High-decision/Low-build` are fine).
>    - **LLM capacity** (agent) — who should build it: 🧠 capable/expensive (judgment, design, prompt-quality, architecture) · ⚙️ mid · 🔧 simple/mechanical, low-decision.
>    - **Lane** — `· Lane: <name>` where the phase runs parallel work lanes; it names the code territory the item lives in (see Phase 6 → _Parallel work lanes_).
>
>    The annotation is **required on every open item and living** — re-rate it whenever work touches the item — and **dropped** when the item is checked `[x]`.
>
> Not every item needs all eight — a trivial mechanical fix can stay terse — but the **default** is the fuller Phase-6 form, and a completed item **keeps** its record inline (see Phase 6) rather than being deleted. Keep any multi-line notes or example lines inside a blockquote (like this one) so the routing-annotation contract doesn't read them as milestones.

### Parallel work lanes

> Phase-6 work is split into code **territories** ("lanes") so multiple agents/sessions can run concurrently without colliding on a hub file. **This axis is orthogonal to the thematic sub-headers below** (experience / signal / validation): a lane is _where the code lives_, a sub-header is _what value it delivers_. Each open milestone carries a `· Lane: <name>` tag on its routing annotation. Two milestones are parallel-safe **iff they're in different lanes and neither is Visual.** Branch discipline for concurrent work lives in [CLAUDE.md](../CLAUDE.md) → _Working alongside parallel sessions_.

**Hub files (single-writer bottlenecks — a lane "owns" the ones it edits):** `src/sidecar/SidecarFeed.tsx`, `src/styles.css`, `src/services/evaluator.ts`, `src/services/evaluatorPrompts.ts`, `src/editor/Editor.tsx`, `src/services/orchestrator.ts`.

> **Rules of the road** (in a blockquote so the routing-annotation test doesn't read these as milestones):
>
> - The four cleanly-concurrent lanes are **Editor · Feed UI · Prompt/signal · Validation** — one item from each = 4 parallel sessions with no shared hub.
> - **Lifecycle shares `evaluator.ts` with Prompt/signal** — parallel-safe only if edits stay in different functions (Lifecycle → `orchestrator.ts` + the emit/cache paths; Prompt/signal → `evaluateSection` + prompt text). Coordinate if both are active at once.
> - **Visual runs solo.** It rewrites `styles.css` wholesale and touches class names across the feed and editor, so it collides with any UI lane — sequence it before or after UI work, never alongside.
> - **Maturity proxy (R2)** starts as a _new pure module_ + `priority.ts`; that first slice is parallel-safe with almost anything and only enters the Lifecycle/Feed hubs once it wires severity/voice into the pipeline.
> - **Shared low-churn files** — `src/services/types.ts` (append-only-ish), `src/styles.css`, and any `db.ts` schema-version bump are touched by several lanes; keep edits minimal and let only **one lane bump the DB version** at a time.
> - **Serialize _within_ a lane:** items sharing a hub file (all Feed-UI items on `SidecarFeed.tsx`, all Prompt items on `evaluatorPrompts.ts`) cannot run concurrently with each other.
