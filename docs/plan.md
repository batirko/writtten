> **Routing legend.** Open milestones below carry an annotation `вЂ” <readiness> <complexity> В· <agent>` so it's clear what's ready, how hard, and who should build it. Completed (`[x]`) lines are left unannotated.
>
> - **Readiness:** рџџў fully defined, ready to build В· рџџЎ mostly defined, decisions along the way В· рџџ  not defined, needs pre-work/planning В· рџ”ґ concept only, no design.
> - **Complexity:** Low В· Med В· High.
> - **Agent:** рџ§  capable/expensive (judgment, design, prompt-quality, architecture) В· вљ™пёЏ mid В· рџ”§ simple/mechanical (well-specified, low-decision).
>
> **This is required metadata, not decoration.** Every new open milestone (here or in any phase section) gets the annotation **on creation** вЂ” there is no unannotated open item. And it's **living:** whenever work touches an item, re-assess and update its annotation in the same change. **Readiness** moves the most (e.g. a design spec lands в†’ рџџ в†’рџџЎ, or a milestone is fully specced в†’ рџџЎв†’рџџў); **complexity** and **agent** shift less often but get re-rated when new information changes the picture (e.g. a "simple" item turns out to need architectural judgment в†’ рџ”§в†’рџ§ ). When an item is completed, drop the annotation as the `[x]` is added.

> **Milestone anatomy вЂ” the default entry style.** The **Phase 6** entries (now in [`plan-archive.md`](plan-archive.md)) are the template: a plan item is a **living record of a decision**, not a one-line to-do. Write new milestones вЂ” and grow existing ones as work lands вЂ” from as many of these parts as apply, in this order:
>
> 1. **Bold title**, carrying any tracking id(s): `**Widen the section check to same-section pairs (OBS-033/UX-018)**`.
> 2. **Provenance** вЂ” how and when it surfaced, as an **absolute date**: `field-discovered 2026-07-08`, `decided 2026-06-18 (with the user)`, `field-observed (owner)`.
> 3. **The problem** вЂ” what's wrong and why it matters, concretely: name the real session/example that exposed it and the `file.ts:line` if known.
> 4. **The decision** вЂ” a **bold state marker + date** naming the chosen mechanism (and the instructive rejected alternative): `**Decided 2026-07-09 (mechanism A):** вЂ¦`. The markers trace an arc a live entry walks: **Field-discovered в†’ Decided / Design settled в†’ Shipped**.
> 5. **Implementation specifics** вЂ” the exact files, functions, and tokens in `code`, so the entry is self-locating without opening the PR.
> 6. **Verification** вЂ” the guard that keeps it fixed: the ratchet fixture, unit test, or live-verify.
> 7. **Cross-references** вЂ” trailing `в†’ see docs/projects/<file>.md` (the design home) and `В· resolves OBS-NNN / UX-NNN`.
> 8. **Routing annotation** (open items only) вЂ” the four routing axes defined by the legend above, appended as `вЂ” <readiness> <complexity> В· <agent>` plus, in a parallel-lane phase, `В· Lane: <name>`. Spell out all four:
>    - **Readiness** вЂ” how defined / ready-to-build the work is: рџџў ready В· рџџЎ mostly, decisions along the way В· рџџ  needs pre-work/planning В· рџ”ґ concept only.
>    - **Estimated size** (complexity) вЂ” Low В· Med В· High (compound forms like `MedвЂ“High` or `High-decision/Low-build` are fine).
>    - **LLM capacity** (agent) вЂ” who should build it: рџ§  capable/expensive (judgment, design, prompt-quality, architecture) В· вљ™пёЏ mid В· рџ”§ simple/mechanical, low-decision.
>    - **Lane** вЂ” `В· Lane: <name>`; it names the code territory the item lives in (see the _Parallel work lanes (live)_ legend below; the original Phase-6 lane table + rules of the road are in [`plan-archive.md`](plan-archive.md)).
>
>    The annotation is **required on every open item and living** вЂ” re-rate it whenever work touches the item вЂ” and **dropped** when the item is checked `[x]`.
>
> Not every item needs all eight вЂ” a trivial mechanical fix can stay terse вЂ” but the **default** is the fuller Phase-6 form, and a completed item **keeps** its record inline (see the archived Phase 6) rather than being deleted. Keep any multi-line notes or example lines inside a blockquote (like this one) so the routing-annotation contract doesn't read them as milestones.

> **Parallel work lanes (live).** The lane axis names the **code territory** an item lives in, so concurrent sessions don't collide on a hub file. The six Phase-6 lanes still apply and keep their definitions in [`plan-archive.md`](plan-archive.md) в†’ _Parallel work lanes_ (**Editor** В· **Feed UI** В· **Prompt/signal** В· **Validation** В· **Lifecycle** В· **Visual**, with the hub-file table and rules of the road). > - **Site/docs** вЂ” README, `docs/`, site legal pages, spec-writing. No `src/`; parallel-safe with everything.
> - **Ops** вЂ” CI workflows, `scripts/`, GitHub settings, the release process. No `src/`; parallel-safe with everything (but only one Ops change to the release path at a time).
> - **Platform** вЂ” app-shell/packaging territory: Tauri wrapper, serverless proxy, external integrations, monetization plumbing. Nothing here is active; re-rate when an item is scheduled.

## Standing rules across all phases

1. **No fix-application affordances. Ever.** (The product principle вЂ” see `CLAUDE.md`.)
2. Observations stay within the **fixed, typed taxonomy**.
3. **No per-keystroke full-document scans;** cross-doc checks go through the claim ledger.
4. **Quiet while generating, opinionated while revising.**
5. **Local-first / privacy** вЂ” no required server, telemetry, or egress without a decision logged in this file.
