> **Routing legend.** Open milestones below carry an annotation `тАФ <readiness> <complexity> ┬╖ <agent>` so it's clear what's ready, how hard, and who should build it. Completed (`[x]`) lines are left unannotated.
>
> - **Readiness:** ЁЯЯв fully defined, ready to build ┬╖ ЁЯЯб mostly defined, decisions along the way ┬╖ ЁЯЯа not defined, needs pre-work/planning ┬╖ ЁЯФ┤ concept only, no design.
> - **Complexity:** Low ┬╖ Med ┬╖ High.
> - **Agent:** ЁЯза capable/expensive (judgment, design, prompt-quality, architecture) ┬╖ тЪЩя╕П mid ┬╖ ЁЯФз simple/mechanical (well-specified, low-decision).
>
> **This is required metadata, not decoration.** Every new open milestone (here or in any phase section) gets the annotation **on creation** тАФ there is no unannotated open item. And it's **living:** whenever work touches an item, re-assess and update its annotation in the same change. **Readiness** moves the most (e.g. a design spec lands тЖТ ЁЯЯатЖТЁЯЯб, or a milestone is fully specced тЖТ ЁЯЯбтЖТЁЯЯв); **complexity** and **agent** shift less often but get re-rated when new information changes the picture (e.g. a "simple" item turns out to need architectural judgment тЖТ ЁЯФзтЖТЁЯза). When an item is completed, drop the annotation as the `[x]` is added.

> **Milestone anatomy тАФ the default entry style.** The **Phase 6** entries (now in [`plan-archive.md`](plan-archive.md)) are the template: a plan item is a **living record of a decision**, not a one-line to-do. Write new milestones тАФ and grow existing ones as work lands тАФ from as many of these parts as apply, in this order:
>
> 1. **Bold title**, carrying any tracking id(s): `**Widen the section check to same-section pairs (OBS-033/UX-018)**`.
> 2. **Provenance** тАФ how and when it surfaced, as an **absolute date**: `field-discovered 2026-07-08`, `decided 2026-06-18 (with the user)`, `field-observed (owner)`.
> 3. **The problem** тАФ what's wrong and why it matters, concretely: name the real session/example that exposed it and the `file.ts:line` if known.
> 4. **The decision** тАФ a **bold state marker + date** naming the chosen mechanism (and the instructive rejected alternative): `**Decided 2026-07-09 (mechanism A):** тАж`. The markers trace an arc a live entry walks: **Field-discovered тЖТ Decided / Design settled тЖТ Shipped**.
> 5. **Implementation specifics** тАФ the exact files, functions, and tokens in `code`, so the entry is self-locating without opening the PR.
> 6. **Verification** тАФ the guard that keeps it fixed: the ratchet fixture, unit test, or live-verify.
> 7. **Cross-references** тАФ trailing `тЖТ see docs/projects/<file>.md` (the design home) and `┬╖ resolves OBS-NNN / UX-NNN`.
> 8. **Routing annotation** (open items only) тАФ the four routing axes defined by the legend above, appended as `тАФ <readiness> <complexity> ┬╖ <agent>` plus, in a parallel-lane phase, `┬╖ Lane: <name>`. Spell out all four:
>    - **Readiness** тАФ how defined / ready-to-build the work is: ЁЯЯв ready ┬╖ ЁЯЯб mostly, decisions along the way ┬╖ ЁЯЯа needs pre-work/planning ┬╖ ЁЯФ┤ concept only.
>    - **Estimated size** (complexity) тАФ Low ┬╖ Med ┬╖ High (compound forms like `MedтАУHigh` or `High-decision/Low-build` are fine).
>    - **LLM capacity** (agent) тАФ who should build it: ЁЯза capable/expensive (judgment, design, prompt-quality, architecture) ┬╖ тЪЩя╕П mid ┬╖ ЁЯФз simple/mechanical, low-decision.
>    - **Lane** тАФ `┬╖ Lane: <name>`; it names the code territory the item lives in (see the _Parallel work lanes (live)_ legend below; the original Phase-6 lane table + rules of the road are in [`plan-archive.md`](plan-archive.md)).
>
>    The annotation is **required on every open item and living** тАФ re-rate it whenever work touches the item тАФ and **dropped** when the item is checked `[x]`.
>
> Not every item needs all eight тАФ a trivial mechanical fix can stay terse тАФ but the **default** is the fuller Phase-6 form, and a completed item **keeps** its record inline (see the archived Phase 6) rather than being deleted. Keep any multi-line notes or example lines inside a blockquote (like this one) so the routing-annotation contract doesn't read them as milestones.

> **Parallel work lanes (live).** The lane axis names the **code territory** an item lives in, so concurrent sessions don't collide on a hub file. The six Phase-6 lanes still apply and keep their definitions in [`plan-archive.md`](plan-archive.md) тЖТ _Parallel work lanes_ (**Editor** ┬╖ **Feed UI** ┬╖ **Prompt/signal** ┬╖ **Validation** ┬╖ **Lifecycle** ┬╖ **Visual**, with the hub-file table and rules of the road). Post-launch work added four territories (2026-07-10):
>
> - **Model/router** тАФ `src/model/*`: provider adapters, pools/cool-down, capability descriptor, debug log. Shares little with the eval lanes; parallel-safe with most.
> - **Site/docs** тАФ README, `docs/`, site legal pages, spec-writing. No `src/`; parallel-safe with everything.
> - **Ops** тАФ CI workflows, `scripts/`, GitHub settings, the release process. No `src/`; parallel-safe with everything (but only one Ops change to the release path at a time).
> - **Platform** тАФ app-shell/packaging territory: Tauri wrapper, serverless proxy, external integrations, monetization plumbing. Nothing here is active; re-rate when an item is scheduled.
>
> Same rules as Phase 6: two items are parallel-safe iff they're in different lanes and neither is Visual; items sharing a lane's hub file serialize within the lane.
>
